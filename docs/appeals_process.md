# 申诉与更正处理流程

**状态：** Draft，MVP 版（静态站架构下的可行版本，非 PRD §30.2 描述的完整自动化处理）
**对应 PRD 条款：** §30（申诉与更正）、PRD v1.2 C7（申诉 SLA）、§36 上线验收条件"删除和申诉机制可用"

## 为什么是这个版本，不是 PRD 原描述的全自动版本

PRD §30.2 设想的是"系统自动检查是否已有相关来源、自动搜索补充证据、自动重新计算事件"——这需要一个能被外部请求触发、能回写数据库的后端服务。当前 MVP 架构是纯静态站（PRD v1.2 C3 的降级决定：无服务器、无数据库对外接口），刻意如此以控制成本和攻击面。

因此本轮实现的是**人工处理、但有明确 SLA 和分级规则的版本**：不是"没有申诉机制"，是"申诉机制现阶段由创始人本人手动处理，而不是自动化流水线处理"，这与 PRD §5.2 "MVP 阶段 Security Administrator 与 Appeals Administrator 均由创始人兼任"的设定一致。

## 可申请事项（PRD §30.1）

- 信息错误（日期/地点/动物类型/伤害方式等字段有误）
- 身份泄露（未成年人或普通个人身份信息被错误公开）
- 事件重复（应合并为同一 Incident 却被拆分为多条）
- 反证未显示（存在但未被记录的矛盾证据）
- 官方回应未收录
- 媒体版权问题
- 其他

## 分级处理（PRD v1.2 C7）

| 申诉类型 | 首次响应 | 处理承诺 |
|---|---|---|
| **高风险**（涉及未成年人身份、住址、法律威胁） | 页面**立即**自动/手动进入临时限制状态（隐藏相关字段或整页下线，视严重程度） | 14 天内完成处理 |
| **一般**（信息错误、事件重复、反证未显示等） | 记录在案，不影响页面现状 | 30 天内完成处理 |

## 内部处理步骤

1. 记录申诉（内部日志，格式待定，参考 `docs/human_review_log.md` 的可追溯记录惯例）
2. 核实：检查是否已有相关来源支持申诉内容；如需要，搜索补充证据
3. 视核实结果：更正字段 / 合并重复事件 / 补充反证 / 说明维持现状的理由
4. 更新 `incidents_public.last_updated_at`；重大更正需在事件页面体现"更正记录"
5. 答复申诉人处理结果

## 已知局限（如实披露）

- 现阶段无法自动检测重复申诉、无法自动核验补充来源的真实性——这些判断依赖创始人本人的人工审核，与整个项目"无需日常逐案审核"的定位存在张力，但申诉处理本身按设计就不是日常逐案审核，而是响应式、按需触发的
- 处理时限承诺（14/30 天）尚未经过实际申诉量的压力测试；MVP 阶段预期申诉量很低（尚未公开发布），承诺的时限届时可能需要重新评估
