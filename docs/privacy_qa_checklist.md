# 隐私 QA 签核清单（HRL-008）

每次真正对外发布数据（不是本地暂存构建）之前过一遍本清单，本人签核。分三部分：AI 已用代码验证、可直接采信的结构性保证；AI 做了扫描但仍需你最终判断的项目；纯人工才能判断、AI 无法代劳的项目。

## 一、结构性保证（代码已验证，2026-07-27）

以下几项是流水线设计本身的硬约束，不是靠"记得检查"，已用代码/查询核实为真，本次不需要你重新验证，只需知道有这道保险：

- [x] Schema 本身不存在姓名/人脸/住址字段——`schema.sql`/`data_dictionary.csv` 里没有任何 name/face/address 列，个人身份信息只可能以自由文本形式出现在 `claim_value`/`summary_zh` 里，不会通过结构化字段意外带出。
- [x] 导出脚本排除内部专用字段——`export_dataset_csvs.py`/`export_site_data.py` 的字段清单里不含 `seed_candidate_no`、`inclusion_note`、`needs_primary_source_verification`（后者本身就是"是否需要人工核实一手来源"的内部审阅备注，混进公开层没有意义还可能误导）。
- [x] 媒体文件不随数据集分发——检查了 `release/ahid-cn-dataset-v0.1/data/*.csv` 与 `site/src/data/incidents.json`，没有任何文件路径、base64 内容或本地快照引用，只有来源 URL 元数据。
- [x] 未成年人事件零身份泄漏——`minor_involvement='yes'` 的全部 4 条事件（#1 揭阳虐狗案、#21 徐州男孩案、#22 连云港男童案、#23 大邑初中生案）逐条读取全部 claim 原文，确认不含任何姓名、面部、学校班级以外的定位信息、家庭住址；`official_response` 字段里出现的最细粒度信息是"辖区九里派出所""大邑县教育局"这类机构级别描述，没有可用于识别具体个人的信息。这是 HRL-015 里唯一不可协商的一条，结果是干净的。

## 二、候选人名扫描（`pipeline/privacy_scan.py`，2026-07-27）

**方法说明，如实披露局限**：这不是真正的中文命名实体识别，是一个粗暴的启发式规则（常见姓氏字 + 1-2 个汉字），刻意追求高召回、能接受高误报——因为让你多花几秒钟排除"张家口"不是人名的成本，远低于一个真名漏检的成本。扫描了全部 30 条事件的 151 条 claim + 34 条 response 共 185 段自由文本。

**结果**：

- 已批准的 3 个实名（范源庆 #6、徐志辉 #9、王照蔚，实际出现在 AHID-CN-2026-0031）全部按预期只出现在各自对应的事件里，没有意外出现在其他事件。
- 131 个候选 token 逐一过了一遍，**没有发现任何未经批准的真实姓名泄漏**。误报全部可解释：
  - 大多数是"警方/官方/校方 + 通报/声明/回应"这类词组因为"方"恰好也是姓氏而被误切分（如"方通报""方声明"）。
  - "熊猫""武汉理工""江苏省"这类机构名/地名因含姓氏字被误报（熊、江、武等也是姓）。
  - "刘海"（#29）经查证是**一只流浪猫的名字**（学生给它起的），不是人名。
  - "龙腾"（#31）是施害者自己选的网名"兴汉龙腾"，不是真实姓名——用网名而非真名，本身就是比"某"更进一步的保护。
- 确认 12 条事件正确使用了"X某某"式匿名化标记（邱某某、崔某某、苏某某、李某某、徐某某、杨某某、胡某某、王某某等）。
- 2 处使用"姓+先生/女士"这类介于实名与"某"之间的称呼（#22 姜女士、#27 吴先生）——这是原始来源本身的表述方式，只透露姓氏不透露全名，符合 HRL-015"跟随信源本身识别程度"的默认规则。
