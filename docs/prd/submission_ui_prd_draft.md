# PRD（草案）：公众事件线索提交 UI（前端 + 后端）

状态：**待批准**（HRL-022）。本文档是提案，不是已批准的开发计划——批准后动工。

## 1. 背景与目标

现有提交入口是 GitHub Issue 表单（HRL-021），对非技术用户门槛过高：普通网民没有 GitHub 账号、不知道 issue 是什么，入口形同虚设。用户（PI）明确要求：一个真正的网页表单 UI，普通人打开就能填，提交后自动同步进 GitHub（作为审核工单）——**与"是否自动收录"无关，收录判断保持人工，HRL-021 的边界不变**。

**目标**：非技术用户 30 秒内完成一次线索提交，零注册、零学习成本；每条提交自动变成一条带标签的 GitHub issue，进入现有的 `screen_submissions.py` 预筛 + 人工审核流程；全程 $0/月。

**非目标**（明确排除在本 PRD 之外）：自动收录/自动发布（HRL-021 已定，不重开）；提交者账号体系/登录；提交后的状态跟踪页面（提交者想追进度可以留邮箱，人工回复——现阶段量小，不值得做自动通知）。

## 2. 推荐架构（方案 A）：静态表单 + 一个 serverless 函数 + GitHub Issues 作为存储

```
用户浏览器
  └─ /submit/ 表单页（Astro 静态页，与现有网站同一仓库、同一风格）
       └─ POST → Vercel serverless 函数（site/api/submit.ts，同仓库部署）
            ├─ 校验：必填字段、URL 格式、蜜罐字段、频率限制
            ├─ 人机验证：Cloudflare Turnstile（免费版，替代 CAPTCHA）
            └─ GitHub API：用 fine-grained PAT 创建 issue（仅 issues:write 权限、仅此一个仓库）
                 └─ 自动打 incident-submission 标签 → 进入现有预筛/人工审核流程
```

**关键设计决定：不引入 Supabase，GitHub Issues 本身就是提交数据库。** 理由：

- GitHub Issues 有结构化字段、标签、开/关状态、评论、完整 API——"工单系统 + 可编程后端存储"要的能力它全有，免费且**永不休眠**。
- 现有 `pipeline/screen_submissions.py` 已经以 GitHub Issues 为数据源，方案 A 下**一行都不用改**。
- Supabase 免费版项目 7 天无活动会自动暂停——对一个低频提交场景（预期每月几条）这是真实的运维负担（要配 keepalive 定时任务续命），引入它得到的唯一好处是"数据在一张 Postgres 表里"，而这个好处现阶段没有对应的需求。
- 少一个系统 = 少一套密钥、少一处同步逻辑、少一个出错点。求职季一个人维护，这条原则前面每个决定都在遵守。

如果未来量大到需要真正的数据库（比如要做提交去重分析、提交者信誉），到时候把 GitHub Issues 的存量数据导入 Supabase 是一个下午的脚本活，现在不做不会锁死以后。

## 3. 备选方案（列出但不推荐）

| 方案 | 内容 | 不推荐原因 |
|---|---|---|
| B：+Supabase | 表单 → serverless → Supabase 表 + GitHub 同步 | 双写两个存储要处理同步失败；免费版 7 天暂停问题；用户 dissertation 项目已占用 2 个免费槽位之一；现阶段收益为零 |
| C：Google 表单桥接 | Google Form → Apps Script → GitHub API | 成本最低（此前已提议），但用户明确要真 UI：表单外观不可控、域名是 google.com、与网站割裂；且中国大陆用户访问 Google 表单不稳定——**这一条对本项目是实质缺陷**，目标提交者恰恰主要在墙内 |
| D：Render 后端 | FastAPI 常驻后端（复用 dissertation 模式） | 免费版 15 分钟休眠、冷启动 30-50 秒——用户点提交要等半分钟，表单场景不可接受；Vercel serverless 无此问题 |

方案 C 的大陆可访问性问题反过来也适用于方案 A 的一部分：**Vercel 默认域名（*.vercel.app）在大陆时常不可达**。缓解：绑定自有域名（HRL-006 正好待决——如果买 ahid-cn.org 绑上，可显著改善），或接受初期部分用户需要科学上网。这是方案 A 最大的已知短板，如实列出。Cloudflare Pages 是备选托管（大陆可达性通常略好，同样免费支持函数），若实测 Vercel 不可达可平移，代码几乎不用改。

## 4. 免费额度与成本核算

| 服务 | 用量预估 | 免费额度 | 结论 |
|---|---|---|---|
| Vercel Hobby（已有账号） | 每月几十次函数调用、<1GB 带宽 | 100GB 带宽/月，函数调用对此量级形同无限 | $0，余量巨大 |
| GitHub API | 每条提交 1 次调用 | 认证后 5000 次/小时 | $0 |
| Cloudflare Turnstile | 每次提交 1 次验证 | 免费无限量 | $0 |
| 域名（可选，HRL-006） | — | — | 约 $10-12/年，非必需但强烈建议（大陆可达性） |

**月度成本：$0。** 唯一可能的开销是域名年费，且那本来就是 HRL-006 的独立决定。Vercel Hobby 条款限"个人非商业用途"——学术开源项目符合。

## 5. 滥用防护（公网可写接口的新增攻击面，如实评估）

- **Turnstile 人机验证**：挡脚本批量提交（免费，用户需要一个 Cloudflare 账号拿 site key，5 分钟）。
- **蜜罐字段 + 提交频率限制**（同 IP 每小时上限）：函数内实现，零依赖。
- **最坏情况兜底**：所有提交只会变成带标签的 GitHub issue，**不触碰数据库、不触碰网站内容**——被垃圾提交淹没的最坏结果是"issue 列表很吵"，删掉即可；不存在数据污染或内容注入风险。PAT 用 fine-grained 类型、仅授 issues:write、仅限本仓库，泄漏时的爆炸半径也只是"能开 issue"。
- 表单字段沿用现有 GitHub 模板的设计，含未成年人身份信息强制确认勾选。

## 6. 依赖与前置条件

1. **网站本身尚未公开部署**（noindex 仍在）。提交 UI 是网站的一个页面，网站不公开则 UI 无人可达。HRL-002 家属风险评估已签核，公开部署的决策障碍已清除，但"是否/何时取消 noindex 并部署"是一个独立决定，建议与本 PRD 一并批准或排序。
2. HRL-019（申诉邮箱）建议同步定下——网站一旦公开，申诉页不能还是"联系方式待定"。
3. HRL-006（域名）非硬性前置，但直接影响大陆可达性（见 §3）。

## 7. 工作量估算

**AI 构建（一个工作会话内完成，约 2-4 小时）**：
- `/submit/` Astro 表单页（风格复用现有 global.css，字段沿用 issue 模板）
- `site/api/submit.ts` serverless 函数（校验 + Turnstile 验证 + GitHub API 建 issue + 蜜罐/限频）
- Vercel 部署配置（vercel.json / adapter 设置，保持站点其余部分纯静态）
- `docs/submission_process.md` 更新（新入口纳入流程图）
- 本地无法完整测试 serverless 函数与 Turnstile 的线上行为，部署后需一次线上冒烟测试（AI 给测试清单，用户或 AI 执行）

**用户操作（约 20-30 分钟，全部是账号层面、AI 无法代劳）**：
1. Vercel：导入 AHID 仓库建项目（root 指向 `site/`）——已有账号，5 分钟
2. GitHub：创建 fine-grained PAT（仅本仓库、仅 issues:write），填进 Vercel 环境变量——5 分钟
3. Cloudflare：注册/登录，建一个 Turnstile widget，把 site key/secret 填进 Vercel——10 分钟
4. （可选）HRL-006 买域名并绑定——另计

## 8. 风险清单

| 风险 | 等级 | 缓解 |
|---|---|---|
| Vercel 默认域名大陆不可达 | 高（针对目标用户群） | 绑自有域名；或平移 Cloudflare Pages |
| 垃圾/恶意提交 | 中 | §5 三层防护；最坏情况仅污染 issue 列表 |
| PAT 泄漏 | 低 | fine-grained 最小权限；只存在 Vercel 环境变量，不进仓库 |
| Vercel Hobby 条款变动 | 低 | 架构可平移 Cloudflare/Netlify，无锁定 |
| 提交量长期为零 | 中 | 与 UI 无关——取决于网站流量与推广，如实预期：初期提交量可能就是零，本功能是基础设施投资不是增长引擎 |
