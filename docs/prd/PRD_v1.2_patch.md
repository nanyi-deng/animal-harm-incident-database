## C1 全局命名变更

**原条款：** 产品名 China Animal Harm Open-Source Monitor；示例数据包名 "cahid-dataset"；示例 incident_id 形如 `CHN-GD-JY-2026-0001`（编码省市信息）。

**修改决定：**

1. 项目与系统名：**Animal Harm Incident Database（AHID）**；中文名**动物伤害事件数据库**。
2. 首期语料：**AHID-CN（Chinese-language Incident Corpus）**。地理范围从机构名下沉到语料层；未来扩展沿用 AHID-XX 区域码。
3. 数据包命名：`ahid-cn-dataset-vX.Y`。
4. `incident_id` 规则：`AHID-CN-YYYY-NNNN`。**ID 不再编码省市**——地理归属可能被更正（AF 状态的常见情形），而 ID 必须终身稳定；省市只存在于可更正的字段中。
5. 原文所有 CAHOM / "China Animal Harm Open-Source Monitor" / "cahid" 表述视为已替换。

**修改原因：** （a）与 AI Incident Database（AIID）的命名结构平行，明示方法论谱系；（b）机构名去掉国名与 "Monitor"，降低被读作境外监督机构的政治色彩，此点与 §41 的风险评估一致；（c）解决 v1.0 中 CAHOM 与 cahid 的命名不一致。

**影响范围：** 全文；§32 成功指标实质不变。

## C2 新增第 41 章：法律、合规与创始人风险

**原状况：** v1.0 仅有一句"抓取必须遵守适用法律、平台访问限制和技术规则"，无独立章节。

**修改决定：** 新增第 41 章，内容如下。

### 41.1 采集与存储合规

- 每个平台 Adapter 的配置中增加 `legal_review_status` 字段；任何平台从 Assumed 升级为 Validated 前，必须完成一次该平台 ToS 与 robots 规则的书面检查记录。
- 涉及数据跨境的问题（中国平台内容存储于海外服务器）在正式公开上线前须获得一次法律咨询意见；在此之前系统只做内部存档，不做公开发布。

### 41.2 政治敏感性与创始人安全

本系统客观上记录中国政府机构对事件的回应与不回应，具有政治敏感性。以下三项为**上线前必须完成的决策**（记录于 human_review_log）：

1. 署名方式：实名运营、机构挂靠（大学/NGO）或团队名义；
2. 数据与网站托管的司法辖区；
3. 创始人及境内亲属的风险评估与预案。

对外定位统一使用"公共利益技术 / 证据基础设施 / 计算社会科学方法"框架；宣传材料不使用"监督中国""对华监测"类表述。

### 41.3 版权与媒体再分发原则（升格）

原 §12.2、§24.3、§26.2 的分散规定升格为一条总原则：

> **公开层（网站、API、数据下载）在任何情况下不重新分发第三方媒体文件。** 公开层只提供：来源元数据、文件 Hash 指纹、存档状态、内容等级与文字描述。媒体副本仅保存于受限内部存档，目的限于证据保存、去重计算与研究核验。

**修改原因：** 法律与安全风险是本项目最大的非技术风险；v1.0 完全未覆盖。

## C3 新增第 42 章：成本与人力假设；MVP 技术栈降级

**原条款：** §10.2 抓取频率（官方源每 1 小时、社媒每 4 小时）；§27 技术栈 Prefect/Airflow + PostgreSQL + OpenSearch + FastAPI + Next.js + CDN/WAF。

**修改决定：**

### 42.1 人力假设

- 构建模式：solo 创始人 + AI 助手。AI 助手承担绝大部分工程与文档工作；创始人只承担审核、收录判断、隐私签核与仅本人可执行的事务（IRB、账号、发布、法律与署名决策）。
- 创始人审核时间设每周上限；与求职材料、dissertation 冲突时，本项目无条件让路。

### 42.2 MVP 技术栈（替代 §27 的 MVP 部分）

| 层 | v1.0 原定 | v1.2 MVP | 升级时点 |
|---|---|---|---|
| Orchestration | Prefect/Airflow | **GitHub Actions 定时任务** | 有资金或合作者后 |
| 数据层 | PostgreSQL | **SQLite（单文件、随 repo 备份）**，Schema 与 Postgres 兼容 | 事件量 >1,000 或多人写入 |
| 搜索 | OpenSearch | **Astro 静态站 + pagefind** | 多语言全文检索需求成熟后 |
| 网站 | Next.js + FastAPI | **Astro 静态生成**（无后端、无攻击面、成本近零） | 需要动态功能（订阅、投稿后端）时 |
| 对象存储 | S3 兼容 | 保留（受限内部桶） | — |

### 42.3 运行频率（替代 §10.2 的 MVP 部分）

MVP 阶段所有来源**每日一次**；v1.0 的分级高频轮询整体推迟至自动发现阶段（Phase 2'）。

**修改原因：** 原架构对应约 5 人团队与持续云支出；创始人已有 Astro 生产经验（个人学术网站），静态优先路径学习成本近零。

## C4 修订 §20–§22：状态与分数的判定关系

**原条款：** §21 状态用定性条件定义（"多个独立来源支持"），§22.1 又用分数阈值（A3 页面 70 分）作发布条件，两套体系的关系未定义；§20.3 扣分可使分数为负；权重无版本管理。

**修改决定：**

1. **状态由规则条件判定，分数只作发布门槛，二者不互相推导。** A0–A4/AX/AF 的判定条件为：独立来源簇数量、权威来源有无、重大反证有无（规则引擎执行）；Evidence Sufficiency Score 达到 §22.1 阈值仅决定"该状态的页面是否公开"。定性条件满足 A3 而分数不足 70 时，事件保持 A3 状态但页面不公开。
2. 分数下限为 0；任何扣分组合不产生负数。
3. 引入 `score_version` 字段；§20.2 权重标注为"初始设定、未经校准"，待 §43 gold-standard 集校准后升版，历史事件记录其计算时所用版本。
4. **来源独立性三值判定（v0 保守规则）：** 相同媒体 Hash → 同一来源簇；存在明确引用链（引用 URL、"据某媒体"、正文高度重合且时间晚于共同源）→ 同一来源簇；其余一律标注 `independence_unknown`，在 §20.2 "独立来源 0–20" 项中按部分分值计（建议每个 unknown 簇按独立簇的 50% 计分）。**禁止二值独立性判定。**

**修改原因：** 独立性是评分中权重最高、又最难自动化的判断，二值分类器的错误会级联污染整个分数体系；双体系关系不明会产生矛盾案例。

## C5 新增第 43 章：系统级人工审计与 Gold-Standard 评估集

**原状况：** §1 声明"无日常逐案审核"，但 §31.2 的错误归并率、错误地点率等指标只能靠人工标注测量，§10.2 的"每月模型和规则质量检查"未定义执行方式——三处互相矛盾。

**修改决定：** 新增第 43 章：

1. **抽样审计：** 每季度随机抽样 ≥30 个事件，人工核对事件聚类、去重判定、字段提取与独立性判定的正确性；precision 估计公开发布于 Data 页面。抽样审计不构成逐案审核，与 §1 原则不冲突——它是 §10.2 质量检查的具体化，也是 §31.2 指标的唯一测量方法。
2. **Gold-standard 评估集：** 审计标注累积为评估集，用途：（a）§20 权重校准与 `score_version` 升版；（b）方法论文的评估表；（c）模型/规则变更时的回归测试。
3. 标注工具复用创始人已有的 human-label-tool（自包含 HTML 标注工具，localStorage 自动保存 + CSV 导出）。
4. 双人标注（创始人 + 一名合作者/学生）时报告标注者间一致性；仅单人时如实标注为单人审计。

**修改原因：** 消除"无人工"与质量指标的矛盾；审计本身就是方法论文的核心数据，属于研究产出而非纯成本。

## C6 修订 §28 / §29.2：人脸术语区分

**原条款：** §28 禁止"自动进行人脸身份识别"；§29.2 要求"自动检测……面部；自动删除或遮挡"。两处表述存在表面矛盾。

**修改决定：** 全文统一术语：

- **Face detection（人脸检测）**：允许，且仅允许用于自动遮挡等保护性处理；
- **Face recognition / identity matching（人脸识别/身份匹配）**：禁止，包括将画面中人脸与任何身份数据库、社交账号或其他事件中的人脸进行匹配。

**修改原因：** 检测与识别的混用会被审稿人与批评者抓住，且模糊了系统最重要的伦理边界之一。

## C7 修订 §30：申诉 SLA

**原条款：** §30.3 高风险申诉"等待安全管理员处理"，无时限；安全管理员角色未落实到人。

**修改决定：**

1. 涉及个人身份、未成年人、住址或法律威胁的申诉：页面**即时**自动进入临时限制状态（自动执行，不等待人工）；人工处理承诺 **14 天**内完成。
2. 一般申诉（信息错误、来源补充、重复合并等）：**30 天**内处理。
3. §5.2 明确：MVP 阶段 Security Administrator 与 Appeals Administrator 均由创始人兼任；SLA 据此设定为可兑现的水平。

## C8 新增第 44 章：研究伦理、IRB 与 Positionality

**原状况：** v1.0 无研究伦理章节。

**修改决定：** 新增第 44 章：

1. **IRB determination：** 在任何数据集公开发布或论文投稿前，向创始人所属机构取得正式 IRB determination（预期结论为 not human subjects research 或 exempt，理由是仅处理公开信息；但因内容涉及可识别个人与未成年人，必须取得正式判定而非自行认定）。判定结果写入 methodology.md。
2. **Positionality 声明：** 方法文档公开声明项目起源于一起具体个案（旺旺事件）的倡议行动，并说明这一起源对收录范围与优先级的潜在影响，以及缓解措施（收录标准前置定义、抽样审计、反证强制展示）。
3. IRB determination 与 positionality 声明列为 §36 上线验收条件的第 13、14 项。

**修改原因：** 学术发表的硬性前提；治理成熟度本身是求职与合作场景中的信誉资产。

## C9 新增第 45 章：存续与落日条款

**修改决定：** 新增第 45 章：

1. 每季度的 Zenodo snapshot（§26.3 版本控制的执行形式）同时是本项目的**正式存续机制**：即使网站与 pipeline 停止运行，已发布的数据版本永久可引用。

**修改原因：** solo 项目必须回答"创始人停止投入后数据怎么办"；答案已内嵌于数据发布策略，只需正式化。

## C10 修订 §23：翻译架构改为模板本地化

**原条款：** §23.3 流程为 Controlled Chinese Summary → Machine Translation → 术语验证 → 回译检查 → 发布，即**每个事件、每种语言各走一遍机器翻译**；§23.5 为此配置了整套逐篇质量检查。2026-07-20 第一轮评审曾建议降为双语首发。

**修改决定：** **六语首发保留**（撤回双语建议），架构改为模板本地化，使绝大多数文本不经过机器翻译：

1. **模板一次性本地化：** §22.3 的四类摘要模板及页面全部固定文案，每种语言一次性翻译定稿（机翻初稿 + 人工核对一遍，见 human_review_log HRL-010），此后复用，不再逐事件翻译。
2. **槽位程序化填充：** 日期与数字用 locale 格式化库生成；地名走人工维护的多语对照表（gazetteer）；受控词表值（§23.4 术语表、动物类别、状态标签）走固定翻译表。
3. **繁体中文：** 由简体经 OpenCC 确定性转换生成，不走机器翻译。
4. **残余自由文本**（如 unresolved_questions 的个别句子）：Google Cloud Translation（v3，启用 glossary 锁定 §23.4 术语）翻译，配**脚本化 QA**——数字/日期一致性正则核对、实体在场检查、否定词清单核对、回译相似度比对。全流程不使用 LLM 翻译。
5. **失败回退：** 任一语言 QA 未通过时，该语言页面回退为显示简体中文 + 英语（保留 §23.5 原回退逻辑）。

**修改原因：** 翻译风险集中于自由叙述文本的机翻（否定词翻反、"涉嫌/确认"语气漂移、日韩语实体错乱）；模板化在架构层面消除这些失败模式，而非靠逐篇检查拦截。成本上 Cloud Translation 约 $20/百万字符，MVP 规模下可忽略。数据集 CSV 本身语言中立，不依赖任何翻译。

## C11 修订 §34/§35 及 v1.1 §11.19：研究版 MVP（2026 求职周期）

**原条款：** §35 实施计划为 Phase 0–3（2+4+4 周）直达公开 Beta；v1.1 §11.16 要求先行两周 Source Access Spike；§32.1 以 90 天 1,000 来源 / 100 事件为成功标准。

**修改决定：** 在 v1.0 的完整愿景之前，插入一个**研究版 MVP（Research MVP）**阶段，作为 2026 求职周期的交付单元：

1. **采集方式：** 仅用 v1.1 定义的 Tier D（URL 驱动）——人工收集 30–50 个已知事件的公开 URL（含旺旺事件），脚本执行归档、去重、字段提取、评分与数据集打包。**不做自动发现。**
2. **推迟项：** Source Access Spike、全站 Crawler、平台 Adapter 实测，整体推迟至 Phase 2'（2027）；视频深度处理（关键帧检索、视频 Embedding 去重）推迟，v0 对视频只保存副本 + SHA-256 + 基础 pHash。
3. **时间表（替代 §35）：**

| 阶段 | 时间 | 交付 |
|---|---|---|
| 定名与骨架 | 2026-07 下旬 | repo、PRD v1.2、data dictionary、human_review_log |
| 文档与伦理 | 2026-08（求职材料绝对优先） | methodology.md、OSF 项目页、IRB 材料起草、六语模板与术语表初稿 |
| Pilot | 2026-09 | 20 个已知事件跑通全 pipeline；隐私 QA 流程验证 |
| v0.1 冻结 | 2026-10 上旬 | 30–50 事件；`ahid-cn-dataset-v0.1` + Zenodo DOI + CITATION.cff |
| 方法白皮书 | 2026-10 至 11 | SocArXiv preprint（设计、验证、治理） |
| Demo 站 | 2026-12 至 2027-01（视面试情况） | Astro 静态站，六语模板页 |

4. **成功指标（替代 §32.1 的 90 天指标）：** 30–50 个结构化事件；数据集 DOI 发布；方法文档公开；隐私测试（未成年人/身份/地址自动隐藏）全部通过；零身份泄露事故；每个公开 Claim 可追溯至 Source。

**修改原因：** 研究版 MVP 保留 100% 的学术产出（数据集、gold standard、方法论文），砍掉约 80% 的工程风险与全部平台合规不确定性；符合 v1.1 "测试结果优先于原假设"的精神——自动发现能力在验证前本就是 Assumed。

## C12 修订 §21 AF 状态：纠错页防二次传播

## C14 新增第 46 章：托管与大陆可达性（2026-07-21 决策）

**原状况：** C3 §42.2 只定义了技术栈（Astro 静态生成），未涉及部署的地理可达性。用户 2026-07-21 直接提出"网站要方便中国大陆访问"，评审发现这是 PRD 的空白——网站的核心受众之一（大陆研究者、记者、动物福利倡导者）恰恰是最可能遭遇访问障碍的群体。

**调研结论（2026-07-21，实测搜索验证而非凭记忆判断）：**

1. **Vercel**：确认在大陆访问差，延迟增加 3–8 倍，连接常超时（Vercel 官方知识库自述）。
2. **Cloudflare Pages**：状态不确定，GFW 对 Cloudflare 的封锁按具体 IP/SNI 选择性执行，同一时段有的站点可访问、有的不可访问；**不是确认可用，是"部分环境可能不通"**。
3. **GitHub Pages / jsDelivr**：均已被墙（jsDelivr 曾持有 ICP 备案后被吊销，现无法正常使用）。
4. **ICP 备案 + 大陆 CDN（唯一可确认稳定的方案）**：个人备案要求大陆居住地址 + 大陆手机号 + 人脸核验；机构备案要求大陆注册实体。创始人现居美国，此路径实质上不可行，除非借用大陆亲属身份——而这直接触及 §41.2 已标记为"待决"的创始人及境内亲属风险评估，不是单纯的技术选型问题。
5. **香港 CDN 加速层**：香港不适用 ICP 备案要求，可作为中间方案改善访问，但需付费（非免费层），且不保证 100% 可达。

**修改决定：**

1. **托管方案：Cloudflare Pages**，接受"部分大陆网络环境可能无法稳定访问"的已知局限，不追加香港加速层或 ICP 备案。
2. **如实标注**：网站页脚 / About 页注明可达性局限（模板文案示例："本站在部分中国大陆网络环境下可能访问不稳定，如遇到问题可通过 Zenodo 数据集直接获取研究数据"），并提供 Zenodo DOI 作为不依赖网站可达性的备用获取渠道。
3. **决策依据**：本轮（2026 求职季）网站的首要受众是英文求职材料评审人与国际学术界，"演示型网站"优先于"大陆日常可用服务"；ICP 备案带来的政治风险评估成本超过本阶段收益。
4. **重新评估触发条件**：若项目转向以大陆公众提交/大陆媒体合作为主要增长路径（而非本轮的求职展示定位），或找到无需真实身份关联的大陆合规路径，应重新评估 ICP 备案可行性，并入 §41.2 的家属风险评估流程审议，不单独作为托管技术决策处理。

**修改原因：** 可达性问题表面是技术选型，实质与 §41.2 的政治敏感性风险评估同源——"让大陆用户方便访问"与"用真实身份在大陆监管记录中登记这个项目"是同一枚硬币的两面。本阶段明确选择前者的最小成本版本（如实标注局限），后者留待项目定位变化时重新审议，而非现在仓促决定。

**影响范围：** site/ 部署配置；site/DEPLOY.md 增补 Cloudflare Pages 步骤；首页/About 页文案增补可达性声明；§36 上线验收条件可选增补第 15 项（可达性声明已上线）。
